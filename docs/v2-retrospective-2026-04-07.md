# DraftOrbit V2 深度复盘（2026-04-07）

## 1. 目标定位回看（North Star）

V2 目标：
1. **Chat-first 极简 C 端体验**（一句话或少量选项完成从生成到发布）
2. **X 运营闭环**（生成 → 质检 → 人审 → 发布 → 互动）
3. **自动学习能力**（本账号历史 + 外部链接/知识源）
4. **可控成本与毛利**（免费优先 + 质量兜底，毛利≥70%）

## 2. 当前符合度评估

### 2.1 已达标 / 基本达标
- ✅ **V1 工作台已物理下线**，核心入口收敛至 `/chat` + `/settings` + `/pricing`。
- ✅ **Brief-first 选项驱动**：目标/受众/语气/类型/CTA/主题模板可直接生成，无需长文本。
- ✅ **步骤可见推理链路**：HOTSPOT→OUTLINE→DRAFT→HUMANIZE→IMAGE→PACKAGE。
- ✅ **多账号能力可用**：支持绑定附加 X 账号、默认账号、状态管理、路由校验。
- ✅ **Stripe 订阅链路在线**：月/年套餐、订阅状态查询、取消流程可执行。
- ✅ **UAT 脚本化上线链路可跑通**：preflight + release + full UAT。

### 2.2 未完全达标（重点缺口）
- ⚠️ **知识接入可见性不足**：API 已有，用户端入口弱，导致“可用但看不懂”。
- ⚠️ **结果解释不足**：之前结果卡缺少质量分、成本路由、步骤耗时，不利于信任建立。
- ⚠️ **状态指导不连续**：缺少统一“下一步/阻塞原因”可视提示（尤其新用户）。
- ⚠️ **试用与风控表达不足**：发布前人工确认策略需在界面显式强化，避免误发与封号风险。
- ⚠️ **性能体验仍有优化空间**：UAT 显示生成步骤耗时波动较大（120s~240s 级别）。

## 3. 本轮已执行的收敛升级（2026-04-07）

1. **结果可解释性升级**（`ResultCard`）
   - 新增质量评分块（总分 + 可读性/密度/平台适配）
   - 新增模型/成本信息（路由层级、预算占比、保守模式提示）
   - 新增步骤耗时与瓶颈展示（step latency）
   - 新增“下一步建议”动态文案（可发布/需重写/超长建议）

2. **发布风控强化**
   - 发布按钮新增二次确认
   - 超过 280 字时前端阻断直接发布并提示改为 thread

3. **Chat 页面状态指导增强**
   - 拉取 `/v2/ops/dashboard` + `/v2/usage/overview`
   - 展示“下一步动作 + 阻塞原因 + 关键计数（主题/草稿/发布/回复）”

4. **设置页新增知识接入区（可见即用）**
   - Obsidian 路径接入
   - 本地文件批量导入
   - URL 批量导入
   - 支持自动学习触发与结果提示
   - 新增 Style DNA 重建入口

5. **UAT 严格断言修正**
   - 对 `/billing/success`、`/billing/cancel` 静态页放宽“必须捕获 API 调用”的断言，减少误报。

## 4. 关键数据观察（来自生产 UAT）

- 必须链路通过率：**100%**（auth/workspace/generate/publish/reply/dashboard/usage/billing）
- 主要瓶颈：**generate.brief-chain 耗时高**（约 120s~240s，受上游模型响应与重试影响）
- 非阻断告警：
  - 无可取消订阅时 cancel 返回 404（预期业务行为）
  - 静态页 API 断言已在脚本策略层处理

## 5. 下一轮迭代优先级（P0/P1）

### P0（本周）
1. **生成性能分层策略**
   - Trial：高阶模型保留但增加软超时与回退窗口
   - Paid：free-first 命中失败时优先 floor，再高阶兜底
   - 目标：P50 生成耗时下降 30%+

2. **发布安全带**
   - 账号级冷却间隔提示（UI + API）
   - 连续失败/高频发布时引导手动发布模式

3. **知识接入可追踪**
   - 在设置页展示最近学习任务状态（成功/失败/耗时）
   - 让“接入后到底学到了什么”可见

### P1（两周）
1. **Chat 会话资产化**
   - 会话级“来源证据”面板（引用链接/本地文档命中）
   - 会话导出（发布包 + 思路摘要）

2. **互动链路前台化**
   - 在 Chat 内直接看回复候选与审批队列
   - 形成“单入口完成运营闭环”

3. **毛利看板产品化**
   - 把 free 命中率、fallback 率、单次成本、毛利预测放到运营视图

## 6. 结论

V2 已经完成“结构层面的正确转向”（从复杂工作台转为 Chat-first）。
当前最主要问题已从“有没有功能”切换为“**是否足够快、足够清晰、足够可控**”。

因此下一阶段不再做大拆重构，而是围绕：
- 速度（Latency）
- 可解释（Why / Next Action）
- 风控（人工确认 + 限频）
- 成本（free-first + quality fallback）

做连续收敛，这条路线最符合既定定位与毛利目标。
